## 一、总体结论

**作为单管理员、手工运维的小规模转发面板，EMORELAY 的核心链路是真实可用的**：
TCP 直连转发与两跳 TLS 隧道转发均端到端实测成功，流量统计精确（实发 150000 字节，
面板显示 146.48 KB，分毫不差），防呆、审计、凭据安全设计普遍高于同类开源面板的平均水准。

**但以「真实产品」标准衡量，目前更接近「能用的引擎 + 半成品驾驶舱」**：

1. **普通用户角色形同虚设** —— 多用户/配额体系后端齐全，前端没有给普通用户留任何可用界面；
2. **零主动通知** —— 节点掉线、配额超限、账号到期都只能靠管理员自己刷页面发现；
3. **前端零自动刷新** —— 连「等待节点上线」这种最基本的场景都需要手动刷新；
4. 同类产品标配的负载均衡/故障转移、图表趋势、DNS 重解析、2FA 等均缺失。

## 二、实测确认可用的能力（亮点）

| 能力 | 实测结果 |
|---|---|
| TCP 端口转发 | 客户端 → :20000 → Agent → echo server，50×3000B 往返成功 |
| 两跳 TLS 隧道 | entry(node-a) → TLS → exit(node-b) → 目标，30×1700B 往返成功 |
| 流量统计 | 60s 桶聚合，数值与实发字节精确一致；规则/节点/用户三级聚合 |
| 端口自动分配 | 留空监听端口自动取池内最小可用（20000、20001 递增）|
| 端口池/保留端口校验 | 池外端口被拒，错误显示在表单内 |
| 删除防呆 | 有活跃规则的节点、有关联规则的隧道、最后一个 admin 均拒绝删除 |
| 凭据安全 | token/私钥仅创建时一次性显示；吊销/轮换入口存在 |
| 审计日志 | 全部写操作落库且 UI 可见（设置页最近 50 条 + 规则详情页操作记录）|
| 隧道表单防呆 | 选隧道后规则节点自动锁定为入口节点，端口池徽章联动 |

## 三、缺失/待完善功能（按严重程度）

### P0 — 体系级断裂（功能链路走不通）

#### 1. 普通用户没有可用的前端（最严重）
实测 `testuser` 登录后：
- 侧边栏仍显示全部 7 个导航项（不按角色过滤）；
- 概览/节点/隧道/用户/限速/设置页主区只渲染一个裸英文单词 **`forbidden`**（截图 `03-normaluser-view.png`）；
- 规则页能打开，但 `/api/nodes` 对普通用户 403 → 节点下拉为空 → **「新增规则」按钮被禁用**，
  而空状态文案还在引导「点击右上角『新增规则』开始」；
- 后端明确支持用户自建规则（`crates/panel-server/src/routes/rules.rs:239`
  「普通用户可以为自己创建规则;rule.user_id 设为 claims.sub」），前端链路断裂；
- 反向也不通：**admin 建规则时无法指定归属用户**（表单无用户字段，
  `CreateRuleRequest` 也不接受 user_id）。

后果：P2 交付的「用户到期 / 30 天流量配额 / 超额停用名下规则」整套体系
**在 UI 上没有任何可走通的使用路径**——规则永远只属于创建它的 admin。
这是与同类产品（均有用户自助页）差距最大的一项。

修复方向（按成本递增任选）：
a) admin 建规则表单加「归属用户」下拉（后端加可选 user_id，仅 admin 可传）；
b) 普通用户视角：按角色渲染导航 + 用户版概览（自己的规则/用量/到期）+
   开放只读节点列表（`GET /api/nodes` 放行普通用户或提供精简版端点）让自建规则可用。

#### 2. 零通知/告警渠道
节点掉线、用户超额停用、账号到期、Agent 注册失败——全部只能靠管理员主动刷页面。
无 Telegram / 邮件 / webhook 任何一种渠道（代码无 notification 相关表/模块）。
对「分布在各地的节点 + 收费用户」的真实运营场景，这是仅次于 #1 的缺口。
建议从最廉价的 **webhook（POST JSON）** 起步：节点 online→offline 沿、配额触发停用、
到期停用三个事件即可覆盖 80% 需求。

#### 3. `stats_retention_days` 是摆设，时序表无限膨胀
设置页可配「统计保留天数」（默认 30），但全仓库只有读写校验
（`crates/panel-server/src/routes/system.rs:232,303`），**没有任何后台任务引用它做清理**
——设置页说明文字自己也承认「后续可加清理任务」。`rule_stats` / `node_stats`
按分钟桶持续写入，长期运行必然把 SQLite 写爆，违背 CLAUDE.md「不能把库写爆」红线的本意。
修复成本低：加一个与 user sweeper 同构的定时 DELETE 任务即可。

### P1 — 同类产品标配、本项目缺失

| # | 功能 | 现状与证据 | 真实场景影响 |
|---|---|---|---|

| 5 | 流量趋势图表 | Dashboard 只有数字卡片；详情页 sparkline 无坐标轴（见 P2-10）| 无法回答「昨晚峰值多少」「哪个用户吃流量」 |
| 6 | 域名目标定期重解析 | Agent 仅在 dial 时解析一次，无周期刷新 | 目标用 DDNS 域名时 IP 变更后规则静默断流 |

| 9 | 规则级连接数限制 | TCP relay 仅计数不限制（`relay/tcp.rs:63`）| 单用户可耗尽节点资源 |
| 10 | IP 黑白名单 / ACL | 无 | 防滥用基本手段缺失 |
| 11 | 数据备份/恢复 | 仅规则 JSON 导入导出；无整库备份入口 | 面板机故障 = 全部配置依赖手工 SQLite 文件操作 |
| 12 | Agent 版本管理/升级 | 审计里有 `version=0.1.0`，但节点列表无版本列、无升级提示/通道 | 节点多了之后升级全靠手工 SSH |
| 13 | i18n | 全中文硬编码，无框架 | 面向开源社区的硬伤 |

### P2 — 实测发现的 UX 摩擦（按用户旅程顺序）

1. **全前端零自动刷新（影响最广）**：`web/src` 无任何 `setInterval`（grep 证实，唯一
   setTimeout 是 toast）。实测创建节点后 20+ 秒页面零网络请求，Agent 已上线仍显示
   「从未上线」，必须手动刷新。受影响场景：等节点上线、等隧道 up、看流量增长、
   节点掉线发现。建议至少给节点/概览/隧道状态加 10-30s 轮询。
2. **Dashboard 流量口径自相矛盾**：「总流量 392.58 KB」（规则转发流量）与
   「过去 24h 流量 143.25 MB」（含节点网卡系统流量）并排显示、相差 365 倍、无任何
   标注（截图 `04-dashboard-final.png`）。节点列表「流量」列同样是网卡口径
   （本机实测显示 ↑32.50 MB，实际转发仅 ~300KB）。需统一口径或明确标注「转发流量/系统流量」。
3. **凭据 Modal 流程死路**：PANEL_PUBLIC_BASE_URL 未配时提示
   「请先到『设置』配 Agent 上报端点，**再回到这里**复制安装命令」——但该 Modal
   关闭即永久消失（凭据一次性），「回到这里」不存在，新手会因此丢凭据被迫轮换。
   建议：设置未配时改为引导「先配置再建节点」，或允许节点详情页重新生成安装命令
   （不含私钥部分）。
4. **删除确认弹窗与后端行为矛盾**：节点删除弹窗说「该节点上的规则将无法继续下发，请确认」、
   隧道删除弹窗说「关联规则将失去隧道绑定，请确认」——读起来都是「会删，有副作用」，
   实际点确认后均被后端拒绝（节点：中文错误；隧道：英文错误）。用户白走两步还被文案误导。
   建议弹窗打开时预检并直接显示「有 N 条规则引用，无法删除」。
5. **后端错误透传且中英混杂**：同为校验错误，「节点上仍有活跃规则,请先删除: #1(echo-tcp)」
   是中文，「bad request: listen_port 22 outside node's port pool [20000-20100]」
   「tunnel is referenced by 1 active rule(s); detach them first」是英文带 `bad request:` 前缀。
   需要统一错误信息语言与展示格式。
6. **「搜索当前页」是陷阱**：节点/用户页搜索框只过滤当前分页的 20 条
   （placeholder 已诚实标注「搜索当前页」），数据过百后用户将搜不到明明存在的记录；
   规则页是真·服务端搜索，行为不一致。建议节点/用户也走服务端搜索。
7. **规则列表无隧道标记**：直连规则与隧道规则两行显示完全同构（实测
   echo-tcp / echo-via-tunnel 无法区分），需要点进详情才知道。建议加隧道徽章列。
8. **规则缺少 Agent 侧运行状态**：列表/详情只有配置层「启用/禁用」，Agent apply
   失败（如端口被占）只能从详情页错误曲线/审计推断。建议显示 desired vs actual 状态。
9. **sparkline 误导**：单数据点被渲染成铺满时间轴的实心三角形（截图
   `02-rule-detail.png`），且无 Y 轴刻度、无 X 轴时间、窗口表述为「过去 ~2.4 小时」。
   数据少时观感是「流量持续了几小时」。
10. **30d 用量缓存延迟造成同屏矛盾**：用户页 admin 行「累计流量 292.97 KB」与
    「30d 用量 0 B」同屏（缓存 5 分钟 sweep 才更新），数据可信度打折。建议读取时
    现算或标注「N 分钟前更新」。
11. **到期时间要求 UTC**：表单标注「到期时间 (UTC)」，管理员需自己换算时区，
    且无日期选择器快捷项（+7天/+30天）。
12. 移动端小问题：表格可横向滚动但无视觉提示（右缘无渐隐/阴影）；概览节点卡名称
    被截断为「loca...」；汉堡抽屉打开动画期间截图有遮挡（功能正常）。
13. 顶栏用户名与角色徽章、审计条目的 action 与对象在无障碍树中粘连
    （「adminadmin」「auth.loginuser#1」），屏幕阅读器朗读会出错；视觉无碍。
